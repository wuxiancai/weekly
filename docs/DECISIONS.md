# 架构决策

## 0001 使用 FastAPI + SQLite

第一版使用 FastAPI 提供 Web/API，SQLite 存储 candles、backtest runs、trades、optimization results。

原因：

- 单机可直接运行，适合快速验证策略闭环。
- SQLite 足够存储周线/日线/分钟线的第一版历史与回测结果。
- 后续可以迁移到 PostgreSQL/TimescaleDB，但当前不需要过早引入部署复杂度。

## 0002 回测与模拟策略共享同一指标和信号函数

指标计算、信号生成和风控参数集中在 `app/strategy.py` 与 `app/backtest.py`。

原因：

- 防止页面、回测、实时模拟使用不同逻辑导致结果不可解释。
- 参数优化直接复用真实策略配置。

## 0003 默认优化目标以收益率排序，同时展示风险指标

优化结果按最终收益率排序，但页面同时展示最大回撤、胜率、交易次数和收益回撤比。

原因：

- 用户目标是最大化收益率。
- 交易系统不能只看收益率，否则容易选出过拟合或回撤不可接受的组合。

## 0004 回测使用下一根 K 线开盘成交

策略信号在某根 K 线收盘后才可确认，因此只能在下一根 K 线开盘成交。止盈止损只作用于已经在当前 K 线开盘前存在的仓位，或在当前 K 线开盘刚建立后的仓位。

原因：

- 防止用当前 K 线收盘、最高价、最低价提前决定当前 K 线内的入场。
- 周线级别尤其容易被未来函数夸大收益，必须显式保守撮合。

## 0005 参数优化必须提供 Walk-forward 样本外验证

整段历史优化仍保留用于探索，但新增 Walk-forward：训练段选参，测试段只验证。

原因：

- 最大化收益率很容易过拟合。
- 样本外结果能更真实地暴露参数组合在未知行情上的稳定性。

## 0006 周线止盈止损使用收盘确认

默认周期是 `1w`，策略语义是周线趋势确认。止盈止损不使用周内最高价/最低价插针触发，而是使用当前周 K 收盘价确认是否越过止盈/止损线；反向信号仍然来自上一根已收盘 K 线，并在下一根 K 线开盘成交。

原因：

- 周线趋势策略应避免被单周极端下影线或上影线洗出。
- 2024-08-05 周 K 虽然最低价跌破原 ATR 止损线，但收盘收回且周涨 `0.94%`，不应按插针止损处理。
- 这会让回测更贴近“收盘确认趋势”的策略定义，但也意味着止损不是盘中硬止损模型。

## 0007 支持周线趋势内再入场

在 `EMA > MA` 的长期多头趋势中，如果上一根周 K 回调到 EMA 或 MA 下方，后一根周 K 又重新收上 EMA，并且方向指标、RSI 和成交量过滤通过，则允许重新给出 `LONG` 信号。

原因：

- 固定金叉/死叉或单次趋势信号会错过大趋势中的回调后续涨段。
- 止盈后系统本来允许同向再开仓；真正阻塞再入场的是信号层过滤条件，而不是回测撮合层。
- 趋势内再入场用于捕捉“回调后重新站上趋势”的周线结构，但仍保留 RSI、方向指标和成交量过滤，避免在弱趋势里无条件追多。

## 0008 支持动态阶梯止盈

固定 `take_atr` 止盈会在达到阈值后立即平仓，容易过早退出强趋势。新增可选参数 `take_atr_step`、`take_atr_max`、`take_atr_buffer_pct`：当价格按周收盘达到初始 `take_atr` 后，不立即止盈，而是激活保护位；后续周收盘继续走强时，按 ATR 阶梯上移保护位；若收盘跌回保护位，则以 `TRAIL_TAKE_PROFIT` 平仓。

原因：

- 周线趋势中，固定 8ATR 止盈会错过超强主升段。
- 无限持有到反向信号收益可能更高，但回撤会显著扩大。
- 动态阶梯止盈用于在“多吃趋势”和“控制回撤”之间取得折中，并且默认关闭，只有显式设置阶梯参数才启用。
