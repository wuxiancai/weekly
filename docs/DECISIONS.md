# 架构决策

## 0001 使用 FastAPI + SQLite

第一版使用 FastAPI 提供 Web/API，SQLite 存储 candles、backtest runs、trades、optimization results。

原因：

- 单机可直接运行，适合快速验证策略闭环。
- SQLite 足够存储周线/日线/分钟线的第一版历史与回测结果。
- 后续可以迁移到 PostgreSQL/TimescaleDB，但当前不需要过早引入部署复杂度。

## 0002 回测与模拟策略共享同一指标和信号函数

指标计算、信号生成和风控参数集中在 `app/strategy.py` 与 `app/backtest.py`。

原因：

- 防止页面、回测、实时模拟使用不同逻辑导致结果不可解释。
- 参数优化直接复用真实策略配置。

## 0003 默认优化目标以收益率排序，同时展示风险指标

优化结果按最终收益率排序，但页面同时展示最大单笔回撤、胜率、交易次数和收益回撤比。

原因：

- 用户目标是最大化收益率。
- 交易系统不能只看收益率，否则容易选出过拟合或回撤不可接受的组合。

## 0004 回测使用下一根 K 线开盘成交

策略信号在某根 K 线收盘后才可确认，因此只能在下一根 K 线开盘成交。止盈止损只作用于已经在当前 K 线开盘前存在的仓位，或在当前 K 线开盘刚建立后的仓位。

原因：

- 防止用当前 K 线收盘、最高价、最低价提前决定当前 K 线内的入场。
- 周线级别尤其容易被未来函数夸大收益，必须显式保守撮合。

## 0005 参数优化必须提供 Walk-forward 样本外验证

整段历史优化仍保留用于探索，但新增 Walk-forward：训练段选参，测试段只验证。

原因：

- 最大化收益率很容易过拟合。
- 样本外结果能更真实地暴露参数组合在未知行情上的稳定性。

## 0006 周线止盈止损使用收盘确认

默认周期是 `1w`，策略语义是周线趋势确认。止盈止损不使用周内最高价/最低价插针触发，而是使用当前周 K 收盘价确认是否越过止盈/止损线；反向信号仍然来自上一根已收盘 K 线，并在下一根 K 线开盘成交。

原因：

- 周线趋势策略应避免被单周极端下影线或上影线洗出。
- 2024-08-05 周 K 虽然最低价跌破原 ATR 止损线，但收盘收回且周涨 `0.94%`，不应按插针止损处理。
- 这会让回测更贴近“收盘确认趋势”的策略定义，但也意味着止损不是盘中硬止损模型。

## 0007 支持周线趋势内再入场

在 `EMA > MA` 的长期多头趋势中，如果上一根周 K 回调到 EMA 或 MA 下方，后一根周 K 又重新收上 EMA，并且方向指标、RSI 和成交量过滤通过，则允许重新给出 `LONG` 信号。

原因：

- 固定金叉/死叉或单次趋势信号会错过大趋势中的回调后续涨段。
- 止盈后系统本来允许同向再开仓；真正阻塞再入场的是信号层过滤条件，而不是回测撮合层。
- 趋势内再入场用于捕捉“回调后重新站上趋势”的周线结构，但仍保留 RSI、方向指标和成交量过滤，避免在弱趋势里无条件追多。

## 0008 支持动态阶梯止盈

固定 `take_atr` 止盈会在达到阈值后立即平仓，容易过早退出强趋势。新增可选参数 `take_atr_step`、`take_atr_max`、`take_atr_buffer_pct`：当价格按周收盘达到初始 `take_atr` 后，不立即止盈，而是激活保护位；后续周收盘继续走强时，按 ATR 阶梯上移保护位；若收盘跌回保护位，则以 `TRAIL_TAKE_PROFIT` 平仓。

原因：

- 周线趋势中，固定 8ATR 止盈会错过超强主升段。
- 无限持有到反向信号收益可能更高，但回撤会显著扩大。
- 动态阶梯止盈用于在“多吃趋势”和“控制回撤”之间取得折中；当前默认策略已启用动态阶梯止盈，以适配默认周线回测窗口的收益最大化参数。

## 0009 回测使用窗口前历史预热指标

长周期指标不能只用回测窗口内 K 线重新计算，否则窗口开头的 EMA/MA/ATR/RSI/ADX 会缺少预热数据，导致早期信号丢失。API 回测加载结束日前全部本地历史 K 线计算指标，但通过 `start_trading_ms` 限制只能从用户选择的开始日期之后开仓；返回给页面的 K 线和权益曲线仍过滤到用户选择窗口内。

原因：

- 如果只喂窗口内数据，MA40 要等 40 根周 K 才成型，容易错过窗口早期趋势交易。
- 使用窗口前历史做指标预热更接近真实运行态，因为实盘在某个开始日期时已经能看到此前历史 K 线。
- 交易和展示仍受用户选择窗口约束，避免把窗口前交易计入目标区间。

## 0010 回测资金模型支持固定本金、复利和可选杠杆

默认初始本金为 `10000`，默认复利为 `YES`。当复利为 `NO` 时，每笔交易按固定本金计算仓位，不把上一笔收益自动加到下一笔开仓本金；当复利为 `YES` 时，每次开仓都使用当前权益计算仓位，平仓后的权益会作为下一笔交易本金。杠杆参数默认为 `2`；当杠杆为正数时，按该倍率放大名义仓位，盈亏和手续费都按放大后的名义仓位计算。

原因：

- 用户需要能分别查看固定本金和复利模式下的回测结果。
- 默认周线策略按用户最终确认的页面参数使用 `2` 倍杠杆和复利模式。
- 手续费按名义仓位计算，避免杠杆回测低估交易成本。

## 0011 BTCUSDT 按 USD-M / U本位永续合约展示和计算

BTCUSDT 必须按 Binance USDⓈ-M Futures / U本位永续合约理解：USDT 作为保证金和盈亏结算资产，合约数量单位是 BTC。页面必须明确显示 `BTCUSDT U本位永续合约`、`USDT 保证金 / USDT 结算`、`合约数量(BTC)` 和 `收益(USDT)`。

盈亏公式按 Binance 官方 USDⓈ-M Futures PnL 说明：

- 多单收益 = `(出场价 - 入场价) * 合约数量`
- 空单收益 = `(入场价 - 出场价) * 合约数量`
- 手续费另行扣除，当前回测按名义仓位计算开仓和平仓手续费。

原因：

- USDⓈ-M BTCUSDT 是线性合约，不是 COIN-M 币本位反向合约。
- 只写“收益”容易被误解为 BTC 数量收益或现货买币收益。
- 用户需要从页面上直接看出资金、盈亏和开仓本金都以 USDT 计价。

参考：

- Binance 官方 FAQ：How to Calculate Profit and Loss for Futures Contracts，USDⓈ-M Futures contracts (BTCUSDT)。

## 0014 按交易对和周期隔离默认策略参数

页面默认参数不再只依赖首页或 ETH 页面本身，而是按 `symbol + interval` 建立独立配置。当前页面支持 `1w`、`1d`、`4h` 与 `1h` 周期；用户切换周期时，页面会自动应用当前交易对对应周期的默认参数。

当前默认配置：

- `BTCUSDT / 1w`：保持已固化的周线默认参数，不再随日线优化变化。
- `BTCUSDT / 1d`：作为独立日线配置项，使用无杠杆、最大单笔回撤 50% 内收益优先的专门优化参数。
- `BTCUSDT / 4h`：作为独立 4h 配置项，已按当前 4h 收益最高候选设置默认参数。
- `ETHUSDT / 1w`：作为独立周线配置项，当前使用周线口径 `take_atr=7.5`、`take_atr_max=32`。
- `ETHUSDT / 1d`：使用此前 ETH 日线收益率最高默认参数 `take_atr=6.5`、`take_atr_max=24`。
- `ETHUSDT / 4h`：作为独立 4h 配置项，当前先复制 `ETHUSDT / 1d` 默认参数，后续可单独优化。

原因：

- 周线和日线属于不同周期交易策略，应视为独立模块。
- 当周线策略出现问题或需要调整时，不应影响日线页面继续回测；日线优化也不应改动已固化的周线默认值。
- 页面直接在“周期”参数中切换 `1w/1d/4h/1h`，可以更方便地回测同一交易对下不同周期，同时保持参数口径清楚。

## 0016 增加 4h 独立周期默认参数

页面 `周期` 下拉新增 `4h`，并为 `BTCUSDT` 与 `ETHUSDT` 分别增加独立的 `4h` 默认参数槽位。初始版本 4h 先按对应交易对的 1d 默认逻辑和参数初始化：

- `BTCUSDT / 4h`：复制 `BTCUSDT / 1d`，即无杠杆、复利、`EMA8 / MA40`、多头 RSI `50-80`、空头 RSI `0-100`、止损 `1.6 ATR`、动态止盈启动 `13 ATR`、止盈阶梯 `0.75 ATR`、止盈上限 `18 ATR`、量能 `0.75`。
- `ETHUSDT / 4h`：复制 `ETHUSDT / 1d`，即复利、杠杆 `2`、`EMA15 / MA40`、多头 RSI `35-85`、空头 RSI `0-100`、止损 `1.8 ATR`、动态止盈启动 `6.5 ATR`、止盈阶梯 `1.25 ATR`、止盈上限 `24 ATR`、量能 `1`。

本轮仅按 1d 参数逻辑建立 4h 独立模块并做 BTCUSDT 4h 基准回测，不把该结果视为 4h 最优参数。

原因：

- 用户要求 4h 与 1w、1d 一样是完全独立的交易策略模块。
- 先复制 1d 参数可以快速建立可回测基线，同时避免把未优化的 4h 参数误写回周线或日线默认策略。

- 资金：本金 `10000`、复利 `YES`、杠杆 `0`
- 成本：手续费 `0.0005`、滑点 `0.0005`
- 趋势：`EMA8 / MA35`
- 指标周期：`RSI14 / ATR14 / ADX14`
- 过滤：`ADX >= 25`、多头 RSI `50-80`、空头 RSI `0-100`
- 风控：止损 `0.8 ATR`、动态止盈启动 `3.5 ATR`、止盈阶梯 `0.5 ATR`、止盈上限 `8 ATR`、止盈缓冲 `0`
- 量能：`volume_mult = 1.0`

当前本地 BTCUSDT 4h 数据 `2019-09-08` 至 `2026-06-29` 的回测结果：

- 最终资金 `319351.51 USDT`
- 总收益率 `3093.52%`

- 交易 `460` 笔

原因：

- 增强 ADX 和量能过滤、缩短动态止盈上限后，能明显降低交易次数和提高盈利因子；胜率仍不高，说明当前信号结构本质上更依赖盈亏比，而不是高胜率。

## 0018 BTCUSDT 4h 增加趋势/震荡状态切换策略

用户指出 4h 周期大部分时间是震荡行情，单一趋势策略会在震荡段频繁试错。`BTCUSDT / 4h` 默认策略增加状态切换：

- `TREND`：`ADX >= 25` 且 EMA/MA 分离度达到阈值时，继续使用原 EMA/MA 顺势策略、ATR 止损和动态 ATR 止盈。
- `RANGE`：`ADX <= 18` 且布林带宽 `(上轨-下轨)/中轨 <= 0.08` 时，使用布林带 + RSI 均值回归；接近下轨且 RSI <= 30 做多，接近上轨且 RSI >= 65 做空。
- `NEUTRAL`：趋势和震荡都不清楚时不主动开仓。

当前 `BTCUSDT / 4h` 默认参数：

- 趋势：`EMA8 / MA35`
- 趋势过滤：`ADX >= 25`、`trend_ma_gap_min = 0`
- 震荡过滤：`range_adx_max = 18`、`range_bb_width_max = 0.08`
- 震荡 RSI：`range_rsi_low = 30`、`range_rsi_high = 65`
- 风控：止损 `0.8 ATR`、动态止盈启动 `3.5 ATR`、止盈阶梯 `0.5 ATR`、止盈上限 `8 ATR`
- 资金：本金 `10000`、复利 `YES`、杠杆 `0`

## 0019 增加实盘模拟运行态与一键 systemd 部署

新增 Paper Trading 运行态，用一个共享 `1000 USDT` 模拟账户同时运行 `BTCUSDT / 1d`、`BTCUSDT / 4h`、`ETHUSDT / 1d`、`ETHUSDT / 4h` 策略。模拟交易不提交真实订单，只读取 Binance USDⓈ-M Futures K 线，按现有回测同一套指标、信号、ATR 风控、动态止盈和 U本位 PnL 口径计算。

关键约束：

- 首次启动只拉取历史 K 线预热指标，并把最新已收盘 K 线标记为已处理，不把历史信号当成实时模拟交易。
- 后续只处理新的已收盘 K 线；同一 `symbol + interval + open_time` 只处理一次，避免重复开仓、重复平仓或重复写权益曲线。
- 模拟状态持久化到 SQLite：账户、策略进度、持仓、平仓交易、权益曲线和运行日志。
- Web 新增 `/paper` 状态页；后台 runner 由 systemd 托管，浏览器关闭不影响运行。
- `scripts/deploy_one_click.sh` 负责创建虚拟环境、安装依赖、写入并启动 `weekly-web` 与 `weekly-paper` 两个 systemd 服务；Web 服务监听 `0.0.0.0`。

原因：

- 用户已通过回测确认策略可行，需要部署到云服务器进行实盘模拟。
- 4h 策略需要长期运行和状态持久化，不应依赖浏览器会话或手工终端。
- 当前阶段仍是模拟交易，不需要也不应该配置 Binance 真实下单权限。
